# KV2025 Demo Scenario — Design Document

**Date:** 2026-03-15
**Scope:** Realistic demo scenario replaying kommunalvalg 2025 results as a folketingsvalg-style election night
**Status:** Approved

---

## Problem

The existing demo mode scenario uses fully synthetic data generated by `fake_fetcher.py`. This is useful for pipeline testing but looks nothing like a real election. We want a scenario driven by real kommunalvalg 2025 vote counts, replayed in size-sorted waves (smallest afstemningsomraader first, large urban areas last), with party-only results appearing first and personal candidate votes arriving later — mirroring how results actually come in on election night.

---

## Architecture

Two new components in the code repo:

```
valg/scenarios/
  prepare_kv2025.py     ← one-time preparation script
  kv2025.py             ← Scenario definition
  kv2025/
    wave_00/            ← setup: geografi + kandidat-data files
    wave_01/            ← preliminary, smallest AOs (<500 voters)
    wave_02/
    ...
    wave_NN/            ← fintælling waves with candidate votes
```

The data repo grows naturally during playback:

```
valg-data/
  demo/
    kv2025/
      geografi/             ← written once in wave_00
      kandidat-data/        ← written once in wave_00
      valgresultater/       ← result files appear batch by batch
      valgdeltagelse/       ← turnout files appear batch by batch
      partistemmefordeling/ ← updated each step as running totals
```

The git history of `valg-data` is the record of the election night — each commit corresponds to a wave, just as real sync commits would look. The wave subdirectories in the code repo are an implementation detail invisible to `valg-data`.

---

## Data Source

SFTP: `data.valg.dk` — folder `data/kommunalvalg-134-18-11-2025/`

Relevant subfolders:
- `geografi/` — `Kommune-*.json`, `Opstillingskreds-*.json`, `Afstemningsomraade-*.json`
- `kandidat-data/` — one file per kommune: `kandidat-data-Kommunalvalg-{Navn}-*.json`
- `valgresultater/` — one file per AO: `valgresultater-Kommunalvalg-{Kommune}-{AO}-*.json`
- `valgdeltagelse/` — one file per AO, multiple turnout snapshots per file
- `mandatfordeling/` — one file per kommune, contains byrådssize (used for `n_kredsmandater`)

## KV → FV Data Mapping

### Geography

| KV concept | FV concept | Notes |
|---|---|---|
| Kommune | storkreds | `Kode` as ID, `Navn` as name |
| Opstillingskreds | opstillingskreds | linked to storkreds via kommune |
| Afstemningsomraade | afstemningsomraade | linked to opstillingskreds |
| byrådssize (from mandatfordeling) | `n_kredsmandater` | actual council seat count |

### Parties

Only kandidatlister whose `Bogstavbetegnelse` is in the FT party set are kept; all local/unlisted lists are silently dropped.

FT party set (FV2024): `A B C D F I M O V Ø Å`

Party names are derived from the first occurrence of each letter across all kommuner — so "A" consistently maps to "Socialdemokratiet" regardless of kommune.

### Valgresultater — preliminary (no candidate votes)

```json
{
  "Valgresultater": {
    "AfstemningsomraadeId": "702016",
    "Optaellingstype": "Foreløbig",
    "IndenforParti": [
      {"PartiId": "A", "Partistemmer": 247, "Kandidater": []}
    ],
    "KandidaterUdenforParti": []
  }
}
```

`Partistemmer` = KV `Stemmer` (total list votes including personal preferences).

### Valgresultater — final (with candidate votes)

Same structure with `Optaellingstype: "Fintaelling"` and `Kandidater` populated from KV `Kandidater[].Stemmer`.

### Partistemmefordeling

Aggregated per opstillingskreds from all AOs reported so far. Re-written each wave with the running total for each opstillingskreds that has at least one reported AO. This matches how real data arrives — the processor's `INSERT OR IGNORE` with snapshot timestamps handles accumulation correctly.

### Kandidat-data

Written once in wave_00. KV candidates are not tied to a specific opstillingskreds (they run for the whole kommune), so all candidates for a given party in a kommune are assigned to the largest opstillingskreds in that kommune by eligible voters.

## Wave Schedule

All ~2600 afstemningsomraader are sorted by `eligible_voters` ascending. The preparation script buckets them by voter-count thresholds computed from the actual data distribution, targeting this shape:

**Phase 1 — Preliminary (party votes, no candidates), ~12 steps:**

| Step | Voter range | Approx AOs | Cumulative | Interval (1×) | Notes |
|---|---|---|---|---|---|
| 01 | < 500 | ~80 | 3% | 90s | Læsø, Samsø, tiny rural — arrive early |
| 02 | 500–1000 | ~200 | 11% | 75s | Small rural |
| 03 | 1000–1500 | ~300 | 22% | 60s | |
| 04 | 1500–2000 | ~350 | 36% | 50s | |
| 05 | 2000–2500 | ~350 | 49% | 45s | Midnight rush begins |
| 06 | 2500–3000 | ~300 | 61% | 40s | |
| 07 | 3000–3500 | ~280 | 72% | 40s | |
| 08 | 3500–4000 | ~230 | 81% | 45s | Rush tailing off |
| 09 | 4000–5000 | ~250 | 91% | 55s | |
| 10 | 5000–7000 | ~150 | 97% | 70s | Large city AOs |
| 11 | > 7000 | ~80 | 100% | 90s | Copenhagen inner city, Aarhus C |

**Phase 2 — Final/fintælling (with candidate votes), ~6 steps:**

Fintælling overlaps the tail of the preliminary phase — the smallest places finish counting while large urban AOs are still reporting party totals.

| Step | Covers AOs from | Starts after prelim step | Interval (1×) |
|---|---|---|---|
| 12 | prelim steps 01–02 | step 05 | 75s |
| 13 | prelim steps 03–04 | — | 60s |
| 14 | prelim steps 05–06 | — | 55s |
| 15 | prelim steps 07–08 | — | 55s |
| 16 | prelim steps 09–10 | — | 65s |
| 17 | prelim step 11 | — | 90s |

Total: 1 setup + 17 data steps = 18 steps.
At 1× speed: ~19 minutes. At 5×: ~4 minutes. At 60×: ~19 seconds.

Exact thresholds are computed by the preparation script from the actual data so bucket sizes balance evenly.

## Demo Mode Integration

The existing `Step` dataclass gets an optional `write_fn` field:

```python
@dataclass
class Step:
    name: str
    wave: int | None
    setup: bool = False
    process: bool = True
    commit: bool = True
    base_interval_s: float = 60.0
    write_fn: Callable[[Path], list[Path]] | None = None  # NEW
```

If `write_fn` is set, the DemoRunner calls `write_fn(data_repo)` instead of `fake_fetcher.write_wave(wave)`. The synthetic scenario is unchanged.

Because the KV2025 scenario needs `data_repo` to build its steps (to know where to copy files from), `Scenario` gains an optional `steps_factory`:

```python
@dataclass
class Scenario:
    name: str
    description: str
    steps: list[Step] = field(default_factory=list)
    steps_factory: Callable[[Path], list[Step]] | None = None  # NEW
```

`steps_factory` is called once at `DemoRunner.start()` with the resolved `data_repo` path. The built steps are cached on the runner for the duration of the run.

### KV2025 scenario sketch

```python
# valg/scenarios/kv2025.py

WAVE_DIR = Path(__file__).parent / "kv2025"

def _make_steps(data_repo: Path) -> list[Step]:
    dest = data_repo / "demo" / "kv2025"
    steps = []
    for wave_dir in sorted(WAVE_DIR.glob("wave_*")):
        meta = json.loads((wave_dir / "_meta.json").read_text())
        steps.append(Step(
            name=meta["label"],
            wave=None,
            setup=meta.get("setup", False),
            write_fn=lambda d, src=wave_dir, dst=dest: _copy_wave(src, dst),
            base_interval_s=meta["interval_s"],
        ))
    return steps

KV2025_SCENARIO = Scenario(
    name="KV2025 — Kommunalvalg 18. november 2025",
    description="Rigtige stemmeresultater fra kommunalvalget 2025, afspillet som valgaften.",
    steps_factory=_make_steps,
)
```

Each `wave_NN/` directory contains a `_meta.json` (label, interval, phase) plus the actual FV-format JSON files in the real subfolder structure (`valgresultater/`, `valgdeltagelse/`, etc.). `_copy_wave` copies files into the correct subfolders under `dest`.

### Registration

```python
# valg/demo.py  (existing SCENARIOS dict)
from valg.scenarios.kv2025 import KV2025_SCENARIO
```

## Preparation Script

`valg/scenarios/prepare_kv2025.py` — run once manually, output committed to the code repo.

Steps:
1. Connect to SFTP, download all KV2025 files into a local temp cache
2. Parse geografi — build kommune/opstillingskreds/AO hierarchy with eligible voters
3. Parse mandatfordeling — extract byrådssize per kommune
4. Parse kandidat-data — build party registry (FT letters only), assign candidates to largest opstillingskreds per kommune-party
5. Parse valgresultater — build preliminary and final result records per AO
6. Sort AOs by `eligible_voters`, compute bucket thresholds
7. For each wave bucket:
   - Build the set of FV-format JSON files for that batch
   - Write to `valg/scenarios/kv2025/wave_NN/` with real subfolder structure
   - Write `_meta.json` with label, interval, phase
8. Write wave_00 (setup): `Storkreds.json`, `geografi/` files, `kandidat-data/` files

The script is idempotent — re-running regenerates everything from scratch.

## File Naming

Output files follow the existing FV naming conventions exactly:

- `Storkreds.json`
- `geografi/Kommune-{timestamp}.json`, `geografi/Opstillingskreds-{timestamp}.json`, `geografi/Afstemningsomraade-{timestamp}.json`
- `kandidat-data/kandidat-data-Folketingsvalg-KV2025.json`
- `valgresultater/valgresultater-Folketingsvalg-{ao_id}.json`
- `valgdeltagelse/valgdeltagelse-{ao_id}.json`
- `partistemmefordeling/partistemmefordeling-{ok_id}.json`

Election ID: `KV2025`

---

## Out of Scope

- Regionsrådsvalg data (same SFTP folder, ignored for now)
- Valgforbund modelling (KV has election alliances; FV does not — ignored)
- Accurate tillægsmandater calculation (KV uses D'Hondt per kommune, same as FV kredsmandater; the existing calculator will run but results reflect council seats, not Folketing seats)
- Scenario editor in the browser
